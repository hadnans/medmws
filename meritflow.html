📡 Sequence 1: Complete Delay-and-Sum Imaging

main entry: beamform_and_display.m / custom script

This is the primary workflow demonstrated in the README and Getting Started. When you run a typical imaging script, the following internal functions are called in order.

🚀 Execution call chain (core sequence)
1️⃣ load_example_data.m (or user data loading)
   ├─ reads frequencies.csv , antenna_locations.csv , channel_names.csv
   └─ loads S-parameter scans: e.g. B0_P3_p000.csv , background subtraction

2️⃣ Signal preprocessing (manual or via helper)
   └─ rotation / reference subtraction → raw signals matrix

3️⃣ merit.domain.hemisphere() → creates imaging grid points & axes
   ├─ located in +merit/+domain/hemisphere.m
   └─ returns points [Nx3] , axes struct

4️⃣ merit.beamform.get_delays() → computes propagation delays
   ├─ file: +merit/+beamform/get_delays.m
   ├─ uses antenna locations, channel names & relative permittivity (or multi-layer)
   └─ returns delay matrix [num_channels x num_points]

5️⃣ merit.beamform( signals, frequencies, points, delays, beamformer )
   ├─ main imaging function: +merit/beamform.m
   ├─ internally calls the selected beamformer: merit.beamformers.DAS (Delay-And-Sum)
   ├─ also supports DMAS , MUSIC (if available in +merit/+beamformers/)
   └─ returns complex image volume img

6️⃣ merit.visualize.get_slice() → extract 2D slice from 3D volume
   ├─ file: +merit/+visualize/get_slice.m
   └─ uses imaging points and axes to interpolate slice at given plane (e.g., z=35mm)

7️⃣ MATLAB's imagesc() , plot3 or merit.visualize.plot_volume() → final display
📁 Key files invoked in this flow:
+merit/beamform.m
+merit/+domain/hemisphere.m
+merit/+domain/cuboid.m
+merit/+beamform/get_delays.m
+merit/+beamform/calc_distance_delay.m
+merit/+beamformers/DAS.m
+merit/+beamformers/DMAS.m
+merit/+utils/validate_inputs.m
+merit/+visualize/get_slice.m
+merit/+visualize/plot_volume.m
example_data/*.csv
load_example_data.m (helper script)
💡 Parallel/GPU note: merit.beamform() automatically uses GPU arrays if input signals are on GPU, or parallel workers when configured — no code changes required.

🧠 Sequence 2: Custom Domain & Frequency-Dependent Delays

alternative beamformer (DMAS / MUSIC) & dispersive media

MERIT allows to easily swap beamformers, define arbitrary imaging domains, and incorporate multi-layer tissue delays. Execution flow variation:

  • 1. Domain generation → merit.domain.cuboid('x', [-0.05,0.05], 'resolution', 1e-3) (file: +merit/+domain/cuboid.m) instead of hemisphere.
  • 2. Delay calculation with multi‑layer permittivity → merit.beamform.get_delays(..., 'relative_permittivity', [8, 25], 'thickness', [2e-3, 10e-3]) — calls +merit/+beamform/calc_multilayer_delay.m (internal helper).
  • 3. Beamformer selection → pass merit.beamformers.DMAS (Delay-Multiply-And-Sum) or custom handle. Inside +merit/beamform.m, the function beamformer.apply(signals, delays, freq, ...) is invoked.
  • 4. Post-processing metrics → optionally compute SCR (Signal-to-Clutter Ratio) using merit.metrics.scr(img, target_region) defined in +merit/+metrics/scr.m.
  • 5. 3D visualization → merit.visualize.plot_volume(img, points, 'isosurface', 0.4) draws isosurface in a figure.
🔁 Critical internal call graph for this variant:
main_script.m → merit.domain.cuboid → merit.beamform.get_delays (multi‑layer) → merit.beamformers.DMAS/apply() → merit.metrics.scr → merit.visualize.plot_volume

📊 Sequence 3: Signal Inspection & Preprocessing Tools

utilities: plot_mag_and_phase, channel reordering

MERIT includes dedicated functions to analyze raw signals before beamforming. This sequence is often used for debugging or channel quality assessment.

📈 Signal analysis flow
a) plot_mag_and_phase.m (top-level script in repo) — loads signals, plots magnitude & phase across frequencies.
b) Inside, calls: merit.io.read_signals() (if signals stored in S2P or custom), but most uses direct CSV.
c) merit.manage.reorder_channels(channel_names, new_order) → file +merit/+manage/reorder_channels.m.
d) merit.manage.exclude_channels(channel_mask) → exclude noisy channels before beamforming.
e) merit.visualize.plot_signals(signals, frequencies, channels) generates time/frequency plots.

📁 Relevant internal files:
+merit/+manage/reorder_channels.m
+merit/+manage/select_bistatic.m
+merit/+manage/calibrate_signals.m
+merit/+visualize/plot_signals.m
🧪 This pipeline does not generate an image but is essential to prepare data before calling merit.beamform. The calibration step can invoke merit.manage.background_subtraction().

⚡ Sequence 4: GPU Accelerated Beamforming Internals

performance path — automatic code branching

MERIT exploits GPU acceleration without explicit code changes. The underlying functions check input types and route to optimized paths.

  • User code: gpu_signals = gpuArray(signals); then call merit.beamform(gpu_signals, ...).
  • Inside +merit/beamform.m, the function validate_and_cast detects GPU arrays and ensures delays & frequencies are also transferred to GPU if needed.
  • Beamformer execution: merit.beamformers.DAS (or other) uses pagemtimes and element-wise operations that run natively on GPU.
  • Similarly, if parallel pool is enabled, the beamformer can partition points across workers using parfor inside merit.beamform — check internal helper +merit/+internal/parallel_chunk.m.
🧩 GPU/Parallel supporting files:
+merit/beamform.m (lines ~120-180: dispatches to gpuArray path)
+merit/+beamformers/DAS.m (implements vectorized beamforming)
+merit/+internal/validate_gpu_compatibility.m
+merit/+internal/parallel_chunk.m

📏 Sequence 5: Image Quality Metrics & Analysis

evaluation pipeline (SCR, FWHM, localization error)

After image reconstruction, MERIT provides built‑in metrics to quantify image quality. The typical sequence for research-oriented runs:

🧪 Metrics evaluation flow
img = merit.beamform(...)
↓
scr = merit.metrics.scr(img, points, target_idx, clutter_region) → file +merit/+metrics/scr.m
smr = merit.metrics.signal_to_mean(img, target_roi) → +merit/+metrics/smr.m
fwhm = merit.metrics.fwhm(img, points, 'threshold', 0.5) → +merit/+metrics/fwhm.m
Additionally, merit.metrics.localization_error(estimated_peak, true_tumor_location)
↓
Results aggregated and exported via merit.io.export_metrics() (JSON/CSV)
🔍 These metric functions are typically used after beamforming to benchmark algorithm performance. The source files reside in +merit/+metrics/ and share a common interface: metrics.compute(img, points, varargin).

▶️ What happens when you run "beamform_and_display.m" (default demo)

concrete execution trace
  1. Line 1-5: Calls load_example_data.m — reads antenna positions, frequencies, and two scans (background + target). Computes differential signals.
  2. Line 8: merit.domain.hemisphere('radius', 0.07, 'resolution', 0.0025) generates ~ 3k points inside hemispherical breast model.
  3. Line 12: merit.beamform.get_delays(...) with relative permittivity = 8 (breast tissue) → delays for every antenna-channel & focal point.
  4. Line 15: img = abs(merit.beamform(signals, frequencies, points, delays, merit.beamformers.DAS))
       ➡ Inside beamform.m, the signals are reshaped, delay interpolation is performed in frequency domain (phase shift), then DAS coherent summation yields complex image.
  5. Line 18-19: merit.visualize.get_slice + imagesc → displays axial slice at z=35 mm.
  6. Optionally, the script could compute metrics if uncommented.
⚙️ Behind the scenes: The beamforming uses the for or parfor loop over frequencies and points, optimized with vectorized phase rotations. All delays are precomputed to avoid redundant calculations.
🔍 Summary: MERIT follows a clear, configurable pipeline: Data Loading → Domain Definition → Delay Computation → Beamforming (DAS/DMAS/MUSIC) → Visualization/Metrics. Every major step corresponds to a dedicated package (+domain, +beamform, +beamformers). By examining beamform.m and get_delays.m, you’ll see the core numerical implementation. GPU acceleration and parallelism are automatically leveraged based on input data type.
