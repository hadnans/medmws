Uninformed Search — Find the Node (Frontier + Explored Set)
21 Feb 2026 Uninformed Search
▼
Question

Given an undirected graph with nodes A, B, C, D, E, F, starting from Node A, perform an uninformed search (no strategy / no heuristic) to determine if there is a path to node H.

The graph edges are: A–B, A–D, B–C, B–E, C–D, C–F, E–F.

Maintain a frontier and an explored set. Show each step in a table.

Algorithm Explanation

Uninformed (Blind) Search explores nodes without any knowledge of where the goal is. It uses a frontier (nodes seen but not yet processed) and an explored set (nodes already processed).

Key rule: When you expand a node, add its unvisited neighbors to the frontier. The order in the frontier determines what gets processed next (here: the frontier is managed informally — think of it like BFS notation but without strict queue ordering).

For each step: 1. Pick a node from the frontier → "Selected Node" 2. Check: is it the goal? If yes → done 3. Add its unvisited neighbors to frontier 4. Move selected node to Explored set 5. If frontier is empty and goal not found → goal doesn't exist
Step-by-Step Solution
SelectedIs Goal?ExploredNeighborsFrontierEmpty?Comment
ANoAYesB, C, DNoStart. A's neighbors: B, D (and C via B)
DNoA, DYesB, CNoNext: D (from frontier)
CNoA, D, CYesB, E, FNoC's new neighbors: E, F
BNoA, D, C, BYesE, FNoB's unvisited: already in frontier
ENoA, D, C, B, EYesFNoE's unvisited: F already in frontier
FNoA, D, C, B, E, FYes—YesAll nodes explored. H never seen.
Final Answer"Node H doesn't exist" — H is not reachable from A in this graph.
Key Insights

Why "Node H doesn't exist"?

  • The frontier became empty before H was ever encountered
  • An empty frontier means we've exhausted all reachable nodes
  • H is simply not in the graph — the search proves this

Frontier vs. Explored

  • Frontier: Nodes discovered, not yet expanded
  • Explored: Nodes fully processed (all neighbors seen)
  • Together they prevent infinite loops in cyclic graphs

Depth-First Search — Stack-Based Approach
24 Feb 2026 DFS
▼
Question

Undirected graph: A–B, A–D, B–C, B–E, D–E, D–G, E–F, G–H. Start at A, Goal at F.

Use a stack-based DFS. When multiple unvisited neighbors exist, push them in alphabetical order (so reverse-alphabetical is processed first).

Show the stack state, explored nodes, and final path.

Algorithm Explanation

Depth-First Search (DFS) explores as deep as possible along each branch before backtracking. A stack implements this naturally (LIFO = Last In, First Out).

Stack notation: [top → ... → bottom] Each entry is "Node:ParentNode" to track the path When expanding node X with neighbors N1, N2, N3 (alphabetical): → Push N3, N2, N1 (reverse alpha) so N1 pops first

The path is reconstructed by following parent pointers back from goal to start.

Step-by-Step Solution
SelectedGoal?ExploredHas Neighbors?Stack (top→bottom)Empty?Comment
ANoAYes[B:A, D:A]NoA's neighbors: B, D → push D then B (B on top)
DNoA, DYes[B:A, E:D, G:D]NoD's unvisited: E, G → push G then E
GNoA, D, GYes[B:A, E:D, H:G]NoG's unvisited: H → push H
HNoA, D, G, HNo[B:A, E:D]NoH has no unvisited neighbors. Dead end.
ENoA, D, G, H, EYes[B:A, B:E, F:E]NoE's unvisited: B, F → push F then B
FYes!A, D, G, H, E, F—[B:A, B:E]No🎯 Goal found!

Path reconstruction: F←E:D, D←A → follow parent chain: F → E → D → A

Final Answer (path){ A, D, E, F }
Key Insights

Stack vs. Queue

  • Stack (LIFO): DFS — goes deep first
  • Queue (FIFO): BFS — explores level by level
  • Same exploration code, different data structure

Why push in reverse alphabetical?

  • We want A processed before B, B before C, etc.
  • Stack is LIFO, so push Z before A to get A on top
  • Here: push D before B → B pops first

Question

Same graph as Quiz 2 (A–B, A–D, B–C, B–E, D–E, D–G, E–F, G–H). Start at A, Goal at F.

Use BFS with a queue. Enqueue unvisited neighbors in alphabetical order. Find the shortest path.

Algorithm Explanation

Breadth-First Search (BFS) explores all nodes at depth d before any node at depth d+1. It guarantees the shortest path (fewest edges) in unweighted graphs.

Queue notation: [front → ... → back] (FIFO) Each entry "Node:Parent" to reconstruct path later Enqueue neighbors in alphabetical order → A's neighbors B, D → enqueue B first, then D

  • Path length: 3 edges (same here)
  • Not guaranteed shortest
  • Goes deep before wide

Question

Navigate a maze from Start S = (b1) to Goal G = (e6) using Greedy Best-First Search with Manhattan Distance as the heuristic.

At each step, move to the neighbor (L, U, R, D) with the lowest heuristic value. Show all neighbor values each step.

Algorithm Explanation

Greedy BFS always picks the node that looks closest to the goal, using a heuristic function h(n). It's like following a compass — fast but can get stuck.

Manhattan Distance: h(n) = |col_n - col_G| + |row_n - row_G| = |column - 6| + |row - 6| (Goal is at e6) Direction priority on tie: L > U > R > D (Left first)

Important: Greedy BFS does NOT guarantee the optimal path — it can make local detours.

  • "Recovered" steps backtrack from dead ends
  • Greedy has no memory of path cost

Question

Find the path from A to F using A* algorithm. f(n) = g(n) + h(n).

Heuristic values h(n) to Goal G: A=×(start), B=3, C=2, D=3, E=2, F=0, G=4, H=5.

Edge costs: A–B=8, A–D=2, B–C=2, B–E=1, C–F=3, D–E=4, D–G=4, E–F=7, G–H=1, H–E=1.

Algorithm Explanation

A* combines Greedy BFS (heuristic) with Dijkstra (actual cost). It finds the optimal path if h(n) is admissible (never overestimates).

f(n) = g(n) + h(n) g(n) = actual cost from start to n (path cost so far) h(n) = estimated cost from n to goal (heuristic) f(n) = total estimated cost through n Priority Queue: always expand the node with lowest f(n)

Why A* is optimal

  • Uses actual path cost g(n) — not just heuristic
  • Expands cheapest estimated total path first
  • Never misses a cheaper path if h is admissible

Admissibility check

  • h(F)=0 ✓ (at goal, 0 cost)
  • h(n) ≤ true distance for all n ✓
  • This guarantees A* finds optimal solution

Algorithm Explanation

Minimax assumes both players play optimally. MAX (X) tries to maximize the score; MIN (O) tries to minimize it.

Scores: X wins = +1, Draw = 0, X loses = -1 Algorithm: - If terminal state: return score - If MAX's turn: return max(scores of children) - If MIN's turn: return min(scores of children) Evaluate moves 1, 2, 3 in order. X picks max score.
Minimax Tree Evaluation

Move 1 (X plays center):

→ O plays 2: X plays 3 → X wins (+1)

→ O plays 3: X plays 2 → Draw (0) via O filling = min(+1, 0) = 0

Move 1 minimax value = min(+1, 0) = 0

Move 2 (X plays bottom-center):

→ O plays 1: X plays 3 → X wins (+1)

→ O plays 3: X plays 1 → Draw (0)

Move 2 minimax value = min(+1, 0) = 0

Move 3 (X plays bottom-right):

→ O plays 1: X plays 2 → X wins (+1)

→ O plays 2: X plays 1 → X wins (+1)

Move 3 minimax value = min(+1, +1) = +1 ← MAX picks this!

Final Answer (best move, outcome)(3-W) — X plays position 3, outcome is Win

Why position 3 wins?

  • Whatever O does, X can always win
  • Position 3 creates two winning threats simultaneously
  • This is called a "fork" — O can't block both

Minimax depth

  • At most 3 empty cells → depth ≤ 3 levels
  • MAX levels: pick highest child value
  • MIN levels: pick lowest child value

Question

4 houses at: H₁=(2,2), H₂=(3,4), H₃=(7,1), H₄=(9,5). Find the medical facility location that minimizes total Manhattan distance to all houses.

Apply Greedy Hill-Climbing twice: Trial 1 starts at M1=(10,2), Trial 2 starts at M2=(5,5).

Move directions (tie-break): Left (L) → Up (U) → Right (R) → Down (D).

Understanding the metrics

  • Precision=100%: When we say R, we're always right (no FP)
  • Recall=40%: We only find 2 of 5 actual R samples
  • F1=57.1%: Harmonic mean — penalizes imbalance

When to use each metric

  • High precision needed: spam detection (avoid false alarms)
  • High recall needed: cancer screening (miss nothing)
  • F1: when you need to balance both

Question

12 points. Initial centroids: C1=(5,2), C2=(5,3), C3=(5,4). Use k-means with k=3. Iterate until no cluster assignment changes. Use Euclidean distance, tie-break: assign to lowest cluster number.

Algorithm Explanation

K-Means Clustering groups data into k clusters by alternating between assignment and centroid update.

Step 1 — Assignment: For each point: compute d(point, Cᵢ) for all i Assign point to nearest centroid (tie-break: lower index) d = √[(x-xc)² + (y-yc)²] Step 2 — Update Centroids: New Cᵢ = mean of all points currently in cluster i Repeat until no point changes its cluster assignment.
Iteration 1 — Initial Centroids (5,2), (5,3), (5,4)

After assignment: C1={#4(4,2), #8(9,2), #10(2,2), #12(3,1)}, C2={#2(10,3), #3(3,3), #5(8,3)}, C3={#1(7,6), #6(6,5), #7(6,7), #9(9,4), #11(5,6)}

New centroids: C1 = mean(4,9,2,3 | 2,2,2,1) = (4.5, 1.75) C2 = mean(10,3,8 | 3,3,3) = (7.0, 3.0) C3 = mean(7,6,6,9,5 | 6,5,7,4,6) = (6.6, 5.6)
Iteration 2 — Centroids (4.5,1.75), (7,3), (6.6,5.6)

Changed assignments: #3 moves C2→C1, #8 moves C1→C2, #9 moves C3→C2

New: C1={#3,#4,#10,#12}, C2={#2,#5,#8,#9}, C3={#1,#6,#7,#11}

New centroids: C1 = (3+4+2+3)/4, (3+2+2+1)/4 = (3, 2) C2 = (10+8+9+9)/4, (3+3+2+4)/4 = (9, 3) C3 = (7+6+6+5)/4, (6+5+7+6)/4 = (6, 6)

K-Means key properties

  • Sensitive to initialization: Different starting centroids can give different final clusters
  • Converges when: No point changes its cluster between iterations
  • Not guaranteed optimal: May find local minimum — use K-Means++ for better init
  • Euclidean distance assumes spherical, equally-sized clusters
